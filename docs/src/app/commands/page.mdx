Service requests should be identity-first. Pass `serviceName`, `agentName`,
`taskName`, and the target `siteId`, `loginId`, or `targetServiceId`, then let
agent-browser pick or reuse the managed profile and queue CDP control. Use a
specific `profile`, `runtimeProfile`, or `--profile <path>` only for known-login
overrides or bring-your-own-profile workflows.
Recurring managed profiles should also have a retained `profile_readiness`
monitor for the same target identity. Software clients can call
`upsertServiceProfileReadinessMonitor()` or persist the monitor through
`POST /api/service/monitors/<id>`.

Use service request `action: "probe"` for generic identity, account, readiness,
or compact page-state checks against a leased service tab. A probe request
requires a valid `serviceTabHandle`, `timeoutMs`, `maxReturnBytes`, and a
provider-neutral `probe.detectors` recipe. Built-in detector types are
`url_title`, `selector_text`, `evaluate`, and `client_evidence`; clients can
provide selectors or bounded evaluation snippets for a specific website. Add
`probe.recordFreshness` when the result should merge target and account
freshness evidence into the selected service profile.

Use service request `action: "tab_handle_refresh"` when a leased service tab
handle may be stale. The request accepts the previous `serviceTabHandle`, an
optional `desiredUrl`, and `repairPolicy` of `reject_only`,
`reuse_compatible`, or `open_if_missing`. The response returns a refreshed
handle or structured rejection with candidate evidence, so clients can recover
from closed tabs without dropping to raw CDP target inspection.

Use service request `action: "tab_handle_release"` when a client is finished
with a leased service tab. The request accepts the previous
`serviceTabHandle`, marks only that retained tab closed in service state, and
preserves the shared browser process plus session route for other clients.

Use service request `action: "ui_action"` when a client needs a bounded
provider-neutral UI recipe against a valid `serviceTabHandle`. Supported step
types include `find`, `focus`, `fill`, `type`, `select`, `menu_select`,
`click`, `wait`, `clear`, and guarded `dialog`. Clients supply selectors,
labels, and website-specific recipe instructions; agent-browser validates the
handle, timeout, action count, text caps, trace context, per-step page evidence,
and optional diagnostics on failure. Prefer this action over direct CDP
`Input`, `DOM`, or small `Runtime` scripts for repeated generic interactions.

Use service request `action: "network_capture"` when a client needs capped
provider-neutral network response evidence against a valid
`serviceTabHandle`. Recipes can filter by URL pattern, method, resource type,
and status, and must include `maxEvents` plus a bounded timeout. Metadata is
returned by default with headers redacted. Response bodies require
`captureBodies: true` and a positive `maxBodyBytes`; returned body evidence
includes byte counts and truncation metadata.

Use service request `action: "file_transfer"` when a client needs a generic
file input or download capture routine against a valid `serviceTabHandle`.
Recipes can include `upload` with a selector or visible `labelText`, explicit
`files`, `allowedPaths`, and `maxFiles`, and/or `download` with a click
selector, `directory`, `allowedDirectories`, optional expected file name, and
optional `maxBytes`. agent-browser validates path allowlists, caps, trace
context, selected file names where available, download metadata, and optional
diagnostics on failure while clients supply website-specific selectors and
labels.

## Find elements

Semantic locators with actions (`click`, `fill`, `type`, `hover`, `focus`, `check`, `uncheck`, `text`):

```bash
agent-browser find role <role> <action> [value]
agent-browser find text <text> <action>
agent-browser find label <label> <action> [value]
agent-browser find placeholder <ph> <action> [value]
agent-browser find alt <text> <action>
agent-browser find title <text> <action>
agent-browser find testid <id> <action> [value]
agent-browser find first <sel> <action> [value]
agent-browser find last <sel> <action> [value]
agent-browser find nth <n> <sel> <action> [value]
```

Options:

- `--name <name>`: filter role by accessible name
- `--exact`: require exact text match

Role locators use Chrome's accessibility tree and the same browser-computed
role and accessible name authority as snapshots. Names from
<code>aria-labelledby</code>, hidden referenced text, implicit roles,
dynamically mounted elements, and supported iframe or shadow-tree targets are
matched without falling back to raw <code>textContent</code>. Temporary locator
refs are removed after successful and failed actions.

Examples:

```bash
agent-browser find role button click --name "Submit"
agent-browser find label "Email" fill "test@test.com"
agent-browser find alt "Logo" click
agent-browser find first ".item" click
agent-browser find last ".item" text
agent-browser find nth 2 ".card" hover
```

## Clipboard

```bash
agent-browser clipboard read                      # Read text from clipboard
agent-browser clipboard write "Hello, World!"     # Write text to clipboard
agent-browser clipboard copy                      # Copy current selection (Ctrl+C)
agent-browser clipboard paste                     # Paste from clipboard (Ctrl+V)
```

Clipboard reads use a three-second renderer deadline plus a short CDP response
grace period. Empty clipboard text is successful. JSON output reports
<code>text</code>, <code>empty</code>, and <code>clipboardOutcome</code>.
Failures append diagnostic JSON with a stable <code>code</code>, target
<code>recovery</code>, and <code>recommendedAction</code>. Replace only the
affected tab when recovery is <code>replace_tab_required</code>.

Use `agent-browser click <selector> --capture-clipboard-write --json` only when
the click is expected to call `Clipboard.writeText`. The opt-in response returns
the last text argument with a 4096-character cap, truncation metadata, and
restoration status. The original method descriptor is restored before command
completion, including when the click fails. This does not capture
`Clipboard.write`, legacy `execCommand`, or native browser UI.

### Iframe support

Iframes are detected automatically during snapshots. `Iframe` nodes are resolved and their content is
inlined beneath the iframe element in the snapshot output. Refs assigned to elements inside iframes carry
frame context, so `click`, `fill`, and other interactions work without manually switching frames.

```bash
agent-browser snapshot -i
# @e3 [Iframe] "payment-frame"
#   @e4 [input] "Card number"
#   @e5 [button] "Pay"

# Interact directly using refs — no frame switch needed
agent-browser fill @e4 "4111111111111111"
agent-browser click @e5

# Or switch frame context for scoped snapshots
agent-browser frame @e3
agent-browser snapshot -i             # Only elements inside that iframe
agent-browser frame main              # Return to main frame
```

The `frame` command accepts element refs (`@e3`), CSS selectors (`"#my-iframe"`), or frame name/URL.

`service status` reports the service control-plane state, persisted service state from `~/.agent-browser/service/state.json`, and configured service-mode entities from `agent-browser.json` and `~/.agent-browser/config.json` without launching a browser. In text mode, it summarizes profiles, browsers, and sessions with service, agent, task, profile, lease, cleanup, browser linkage, health, and retained observation fields. Configured profiles, sessions, site policies, and providers override entries with the same IDs from persisted state. It refreshes the persisted control-plane snapshot in `state.json` and probes persisted browser records for dead PIDs and unreachable CDP endpoints. Dead local PIDs are marked `process_exited`, unreachable CDP endpoints with a live PID are marked `cdp_disconnected`, unreachable CDP endpoints without a PID are marked `unreachable`, and endpoints that answer health probes but fail target-list discovery are marked `degraded`. Reachable CDP endpoints are queried for live page and webview targets, updating `tabs` and known session/tab relationships in service state. Non-ready browsers close their known tabs during reconciliation so stale tab state does not look active. When reconciliation removes stale session/tab ownership links, it appends a <code>reconciliation</code> event with <code>details.action: "session_tab_ownership_repaired"</code> and the removed relationships. Browser launch, close, and command-time stale-browser detection update the active session's persisted browser health record. Browser records retain the latest non-ready health evidence in `lastHealthObservation`, so service status clients can inspect failure metadata without reconstructing events. Use `service profiles`, `service sessions`, `service browsers`, `service tabs`, `service monitors`, `service site-policies`, `service providers`, and `service challenges` for focused collection views without parsing the full status payload. Active monitor records are checked by the daemon scheduler when due; use <code>service monitors run-due</code>, HTTP <code>POST /api/service/monitors/run-due</code>, MCP <code>service_monitors_run_due</code>, or <code>runDueServiceMonitors()</code> to check due active monitors immediately. Unexpected process-exit health and recovery events include <code>details.processExitCause: "unexpected_process_exit"</code> and <code>details.failureClass: "browser_process_exited"</code>; active local Chrome exits also include <code>processExitDetection</code>, <code>processExitPid</code>, exit code or signal when available, and <code>browserStderrLogPath</code> when Chrome stderr was captured under <code>~/.agent-browser/tmp/chrome-launches/</code>. Browser launch events include <code>browserStderrLogPath</code> when Chrome stderr capture is available. CDP-disconnect health events also include <code>browserStderrLogPath</code> when agent-browser owns the local Chrome process. During close, agent-browser first tries a polite browser shutdown and then force kills owned browser processes if needed; close-generated health events include <code>details.shutdownReasonKind: "operator_requested_close"</code>, <code>details.processExitCause: "operator_requested_close"</code>, and shutdown outcome flags. Polite shutdown failure leaves the browser record <code>degraded</code>, and force-kill failure leaves it <code>faulted</code> with an OS-degraded warning. When a queued command finds the active browser process exited or the CDP connection disconnected, agent-browser records that failure before cleanup and relaunch. Runtime profile and custom profile launches also populate linked service profile and session records, including <code>serviceName</code>, <code>agentName</code>, and <code>taskName</code> when the caller provides them.

Run `pnpm test:service-site-policy-sources-no-launch` to validate that HTTP `GET /api/service/site-policies`, MCP `agent-browser://site-policies`, and `getServiceSitePolicies()` report config, persisted-state, and built-in site-policy source metadata without launching Chrome. Run `pnpm test:service-collections-live` to validate that CLI, HTTP, and MCP expose matching service-owned profile, session, browser, tab, monitor, site-policy, provider, and challenge collections for one live runtime-profile session. Use HTTP `GET /api/service/monitors`, MCP `agent-browser://monitors`, `service monitors`, or `getServiceMonitors()` to inspect retained monitor records. CLI and HTTP monitor collections support state, failed-only, and summary filters for operator triage; MCP keeps the full unfiltered resource. Active monitors are checked by the daemon scheduler when due. Use `service monitors run-due`, HTTP `POST /api/service/monitors/run-due`, MCP `service_monitors_run_due`, or `runDueServiceMonitors()` to check due active monitors immediately. Use `service monitors pause <id>` and `service monitors resume <id>`, HTTP `POST /api/service/monitors/<id>/pause` and `POST /api/service/monitors/<id>/resume` routes, MCP `service_monitor_pause` and `service_monitor_resume`, or `pauseServiceMonitor()` and `resumeServiceMonitor()` to quiet or restore noisy monitors. Use `service monitors triage <id>`, HTTP `POST /api/service/monitors/<id>/triage`, MCP `service_monitor_triage`, or `triageServiceMonitor()` to acknowledge the related monitor incident and clear reviewed failures in one queued operation. Use `service monitors reset <id>`, HTTP `POST /api/service/monitors/<id>/reset-failures`, MCP `service_monitor_reset_failures`, or `resetServiceMonitorFailures()` when only the reviewed failure counter should be cleared while retaining last failure evidence.

Site policies can recommend a browser build and require CDP-free operation for bot-sensitive services. `service site-policies` includes `build` and `cdp_free` fields in text output, and access-plan responses expose the resolved posture through `launchPosture.browserBuild`, `launchPosture.requiresCdpFree`, and `launchPosture.cdpAttachmentAllowed`. The shipped Canva policy defaults to `cdp_free_headed`.

The persisted service state includes a `reconciliation` snapshot with `lastReconciledAt`, `browserCount`, `changedBrowsers`, and `lastError` so operators can confirm when browser-health probes last ran.

The persisted service state also includes bounded audit records for recent control-plane jobs in `jobs`, a derived `incidents` collection grouped by browser or service scope, plus an `events` log with reconciliation summaries, browser health transitions, browser recovery starts, profile lease wait transitions, ownership-repair details, and tab lifecycle changes such as discovered tabs, URL or title changes, and closed tabs. Job records track request action, priority, timestamps, final success or failure, requested remote-headed `displayIsolation`, and error text without storing large command payloads.

`service status --watch` and `service watch` poll the same status path until interrupted. Use `--interval <ms>` to set the polling interval and `--count <n>` for bounded scripts. In JSON mode, each poll is emitted as one JSON response line.

Ordinary status retains all live and referenced tabs and projects at most 50
unreferenced closed-tab rows. <code>closedTabProjection</code> reports the mode,
cap, total, retained, omitted, and deterministic ordering. Use
<code>service status --full-tab-history</code> or HTTP
<code>GET /api/service/status?full-tab-history=true</code> for the complete
response-only diagnostic projection. Status reads do not compact persisted
service state or change routing and stale-handle authority.

`service status` includes <code>retainedDisplayAllocations</code>, a compact classification summary that separates retained historical display-allocation rows from live browser, session, tab, and route-pool readiness. Use its <code>cleanupCommand</code> value for the dry-run before applying display allocation compaction.

Use <code>agent-browser remote-view open &lt;url&gt; --runtime-profile &lt;id&gt; --browser-build stealthcdp_chromium --view-stream-provider rdp_gateway</code> when the operator or a downstream service needs one controllable Guacamole/RDP browser without manually repairing route URLs, desktop binding, Xauthority, or CDP fields. The legacy <code>--provider rdp_gateway</code> spelling remains a compatibility alias for this subcommand, but <code>--view-stream-provider</code> is canonical because <code>--provider</code> means cloud browser provider elsewhere. Global flags may appear before or after <code>remote-view open</code>; <code>--session</code> selects the agent-browser daemon session, while <code>--session-name</code> selects saved browser state and is copied into the remote-view request. The command selects a retained or inline route-pool entry, launches or reuses the remote-headed browser on the selected route display, opens or reuses one same-origin requested tab, focuses it, checks visible browser-window evidence on that route display, checks out the route, reruns final operator-visible proof from the post-checkout route state, and returns dashboard plus external operator URLs. When <code>AGENT_BROWSER_RDP_ROUTE_POOL_JSON</code> is present, the CLI copies that fresh route-pool array into the request unless the caller already supplied <code>--route-pool-entry-json</code>, so readiness output can override stale daemon-retained route records. Repeated opens on the same route return <code>tabAcquisitionDecision</code> and <code>duplicateTargetCleanup</code> evidence so clients can prove the one-liner converged to one intended target. Successful responses include final <code>operatorVisible</code>; clients should require <code>operatorVisible.state="ready"</code> before claiming the handoff is visible to the operator. <code>operatorVisible</code> also includes selected target evidence and route, display, browser, tab, stream, and Guacamole component states. A visible browser with the wrong selected URL reports <code>operatorVisible.state="wrong_tab"</code>. A ready display and tab with no ready Guacamole operator route reports <code>operatorVisible.state="guacamole_route_unavailable"</code>. A selected tab without a CDP target ID reports <code>operatorVisible.state="cdp_target_unavailable"</code>. Retained route-pool metadata that points at a stale route allocation reports <code>operatorVisible.state="stale_route_record"</code>. Responses include <code>browserBuildProof</code> with requested build, selected build, executable path when known, and mismatch state. Likely one-time operator handoffs that omit <code>runtimeProfile</code> use a deterministic <code>managed_one_time</code> task profile so retries reuse the same lane. If the caller passes an unknown runtime profile instead, responses include <code>oneTimeProfileWarning</code> with the requested profile as <code>operator_supplied</code> and a recommended <code>managed_one_time</code> profile id. Dry runs return <code>operatorVisible.state="not_checked"</code>. Add <code>--route-pool-entry-id</code> or <code>--route-pool-entry-json</code> when a caller already copied a specific route from doctor or route-pool readiness output, and use <code>--dry-run</code> to inspect the launch, tab, and checkout commands before starting Chrome. If post-launch setup or final route verification fails, a newly launched browser is closed before the typed error is returned; reused retained browsers keep the browser process and clean up only the opened tab when possible. Treat <code>agent-browser doctor remote-view --json</code> <code>remoteControl.status=ready</code> as the single-route gate, and require <code>manyToMany.status=ready</code> only when the claim is simultaneous multi-viewer operation.

Use <code>pnpm ensure:rdp-guac-postgres -- --apply</code> when route-pool readiness reports a missing Guacamole schema. It starts the user-scoped Guacamole Postgres service when needed, imports the packaged Guacamole schema only into an empty initialized database, forces a checkpoint, and refuses automatic repair when a partial <code>guacamole_*</code> schema suggests a crash or hard stop needs reviewed database recovery. The route-pool setup and existing-user sync scripts run the same guard before they write Guacamole records.

`service reconcile` runs the same persisted browser health and target probes intentionally without requesting a control-plane status snapshot. It updates `state.json`, refreshes live tab records for reachable browser CDP endpoints, refreshes the `reconciliation` snapshot, appends service events, and returns the reconciled service state plus counts for total and changed browser records. The response also includes `remoteViewRepair` counts for remote-view display, route, viewer lease, and controller lease repairs.

`service prune-retained` is dry-run by default for retained inventory cleanup. It previews closed tabs and inert <code>not_started</code> browser records that are safe to remove from service state, then applies the cleanup only when <code>--apply</code> is present. Add <code>--process-exited-browsers</code> only after reviewing failure evidence because those records may explain a crash or shutdown path. Add <code>--orphaned-profiles</code> to include only <code>custom:*</code> profile records that have no retained service references and point at missing ephemeral user-data directories. Add <code>--display-allocations</code> to classify retained display allocation records and prune only apply-safe orphan, stale-route, or historical-placeholder allocations.

Add <code>--released-sessions</code> when released or expired session records should be pruned with their inert linked browser placeholders. Add <code>--abandoned-sessions</code> only after operator review because it applies the same inert-placeholder cleanup to shared or exclusive session leases that may otherwise affect profile allocation. Abandoned shared or exclusive sessions must also have a parseable <code>lastLeaseObservedAt</code> or <code>createdAt</code> older than <code>--abandoned-session-min-age-minutes</code>, which defaults to <code>1440</code>. Dry-runs report skipped abandoned sessions as <code>abandonedSessionsMissingAgeTimestamp</code> or <code>abandonedSessionsTooFresh</code> so operators can distinguish missing age evidence from lease observations that are still inside the minimum age guard. The <code>skippedSummary</code> field reports the top skipped groups by stable prefix after trimming trailing numeric run suffixes, including total and omitted group counts. Text output shows the same top groups for faster retained-state triage.

`service repair-retained` is dry-run by default for safe retained evidence repair. Run it before abandoned-session cleanup when legacy inert session placeholders lack parseable age evidence. With <code>--apply</code>, the command stamps current observation time into <code>lastLeaseObservedAt</code> only for shared or exclusive sessions that have no retained tabs and only inert self-linked <code>not_started</code> browser placeholders. Repaired sessions become too fresh for <code>--abandoned-sessions</code> pruning until the minimum age guard elapses.

`service access-plan` prints the same no-launch recommendation returned by HTTP <code>GET /api/service/access-plan</code> and MCP <code>service_access_plan</code>. Use it before browser work when an operator, agent, or software client needs the selected profile, manual-seeding posture, monitor freshness recommendation, service request availability, and compact <code>browser_build_summary</code> that explains whether stock Chrome, stealth CDP Chromium, or CDP-free headed mode won. Pass <code>--browser-host</code>, <code>--view-stream-provider</code>, <code>--control-input-provider</code>, and <code>--display-isolation</code> when the caller needs the plan to preserve an explicit remote viewport posture.

`service cancel <job-id>` marks a queued or lease-waiting service job cancelled before it dispatches or requests cooperative cancellation for a running job. Running cancellation drops the active service future, records the job as <code>cancelled</code>, and cleans up browser state before the worker accepts more work. Terminal jobs are rejected rather than rewritten. Add <code>--reason &lt;text&gt;</code> to record an operator-readable reason for queued cancellation.

`service retry <browser-id>` explicitly allows one new recovery attempt for a faulted browser. It records a <code>browser_recovery_override</code> event, moves the browser back to a retryable stale health state, and resets retry counting from that override boundary. Add <code>--by &lt;text&gt;</code> and <code>--note &lt;text&gt;</code> for operator auditability. HTTP retry requests accept <code>service-name</code>, <code>agent-name</code>, and <code>task-name</code> query parameters, and MCP <code>service_browser_retry</code> accepts <code>serviceName</code>, <code>agentName</code>, and <code>taskName</code>, so override events appear in filtered service traces.

`service acknowledge <incident-id>` marks a retained incident seen by an operator. Add `--by <text>` to record who acknowledged it and `--note <text>` to persist a short operator note.

`service resolve <incident-id>` marks a retained incident handled while preserving the derived incident record. Add `--by <text>` to record who resolved it and `--note <text>` to persist a resolution note.

Acknowledgement and resolution also append retained service events with `incident_acknowledged` and `incident_resolved` kinds. Incident detail includes those handling events alongside the health and job events that define the grouped incident. Use `service activity <incident-id>` to fetch a normalized chronological timeline for one retained incident without reconstructing it client-side.

The activity response is the canonical agent-facing incident timeline. It returns <code>{"{ incident, activity, count }"}</code>. Each activity item includes <code>id</code>, <code>source</code>, <code>timestamp</code>, <code>kind</code>, <code>title</code>, and <code>message</code>, plus <code>eventId</code> or <code>jobId</code> when it came from a retained event or job. Event and job items include trace context fields such as <code>browserId</code>, <code>profileId</code>, <code>sessionId</code>, <code>serviceName</code>, <code>agentName</code>, and <code>taskName</code> when known, so clients can display one timeline without rejoining raw records. Older retained incidents can include <code>source: "metadata"</code> acknowledgement or resolution items when handling metadata predates retained handling events.

`service trace --service-name <name> --task-name <name>` returns related events, jobs, incidents, and normalized activity in one response. Add `--limit <n>`, `--browser-id <id>`, `--profile-id <id>`, `--session-id <id>`, `--agent-name <name>`, or `--since <timestamp>` to narrow a trace view for one service, agent, task, browser, profile, session, or time window. Use this when clients need a complete service/task timeline without issuing separate jobs, incidents, events, and incident activity requests. The response includes a <code>summary</code> object with compact service, agent, task, browser, profile, and session context rows plus per-context record counts, target identity hints, control-plane modes, lifecycle-only job counts, naming warnings for missing service, agent, or task labels, and <code>summary.contexts[].attention</code> intervention metadata when incidents or missing labels need follow-up. Attention metadata is UI-neutral and uses <code>presentation: "client_decides"</code> so clients choose whether to log, prompt, show a dashboard affordance, or raise another operator channel. <code>summary.contexts[].targetServiceIds</code> lists the normalized target-service, site, and login identity hints observed on retained jobs in that context; text output shows them as <code>targets=...</code>. <code>summary.contexts[].controlPlaneModes</code> lists <code>cdp</code>, <code>cdp_free</code>, or <code>service</code> modes observed on retained jobs, and <code>lifecycleOnlyJobCount</code> counts CDP-free launch and service lifecycle jobs in that context. <code>summary.displayAllocations</code> aggregates requested private, shared, and ambient display allocation across retained jobs in the trace, and each context also reports observed <code>displayAllocations</code> plus <code>unrecordedDisplayAllocationJobCount</code>. <code>summary.profileLeaseWaits</code> provides a per-job profile lease wait rollup with outcome, timing, conflict sessions, and trace labels. Text output includes a <code>Profile lease waits</code> block when that summary or the raw wait events are present. Browser crash recovery traces expose the canonical sequence in `events`: a `browser_health_changed` event with `currentHealth` such as `process_exited` or `cdp_disconnected` and `details.currentReasonKind`, then `browser_recovery_started` with `details.reasonKind`, `details.reason`, `details.attempt`, `details.retryBudget`, `details.nextRetryDelayMs`, and `details.policySource`, then a `browser_health_changed` event with `currentHealth: "ready"` after relaunch. Stale health and recovery events also include <code>details.failureClass</code>, such as <code>browser_process_exited</code>, <code>cdp_unresponsive</code>, <code>cdp_endpoint_unreachable</code>, or <code>target_discovery_failed</code>. Process-exit health and recovery events also include <code>details.processExitCause: "unexpected_process_exit"</code>. Active local Chrome process exits include <code>details.processExitDetection: "local_child_try_wait"</code>, <code>details.processExitPid</code>, <code>details.processExitCode</code> or <code>details.processExitSignal</code> when available, and <code>details.browserStderrLogPath</code> when Chrome stderr was captured under <code>~/.agent-browser/tmp/chrome-launches/</code>. Browser launch events include <code>details.browserStderrLogPath</code> when Chrome stderr capture is available. CDP-disconnect health events also include <code>details.browserStderrLogPath</code> when agent-browser owns the local Chrome process. `details.policySource` reports `default`, `config`, `env`, or `cli` for the retry budget, base backoff, and max backoff values. Operator-requested shutdown health events instead include <code>details.shutdownReasonKind: "operator_requested_close"</code> and <code>details.processExitCause: "operator_requested_close"</code> plus polite-close and force-kill outcome flags, so clients can separate clean or degraded closes from unexpected exits. If the next attempt would exceed the default retry budget, the browser is marked `faulted` and the command fails instead of relaunching. HTTP clients read the same payload from `/api/service/trace`, and MCP clients read it through the `service_trace` tool.

Software clients can call <code>summarizeServiceTraceAttention()</code> from <code>@agent-browser/client/service-observability</code> on a <code>getServiceTrace()</code> response to roll up required operator follow-up, service labeling follow-up, reasons, suggested actions, messages, and affected contexts without copying dashboard logic. Service browser-health reconciliation runs in the daemon background every 60000 ms by default. Reconcile also expires due service session leases without launching a browser and returns <code>expiredSessionLeases</code>, <code>expiredSessionLeaseCount</code>, and <code>remoteViewRepair</code> in the service reconcile response. Set <code>service.reconcileIntervalMs</code>, pass <code>--service-reconcile-interval &lt;ms&gt;</code>, or set <code>AGENT_BROWSER_SERVICE_RECONCILE_INTERVAL_MS</code> to change the interval. Use <code>0</code> to disable it.

Due active service monitors are enqueued through the same service worker every 60000 ms by default. Set <code>service.monitorIntervalMs</code>, pass <code>--service-monitor-interval &lt;ms&gt;</code>, or set <code>AGENT_BROWSER_SERVICE_MONITOR_INTERVAL_MS</code> to change the scheduler interval. Use <code>0</code> to disable monitor scheduling. Use <code>service monitors run-due</code>, HTTP <code>POST /api/service/monitors/run-due</code>, MCP <code>service_monitors_run_due</code>, or <code>runDueServiceMonitors()</code> to check due active monitors immediately. Use monitor pause/resume commands, HTTP <code>POST /api/service/monitors/&lt;id&gt;/pause</code> and <code>POST /api/service/monitors/&lt;id&gt;/resume</code> routes, MCP tools, or client helpers to change only the retained monitor state while preserving health evidence. Use monitor triage commands, HTTP <code>POST /api/service/monitors/&lt;id&gt;/triage</code>, MCP <code>service_monitor_triage</code>, or <code>triageServiceMonitor()</code> to acknowledge the related monitor incident and clear reviewed failures in one queued operation. Use monitor reset commands, HTTP <code>POST /api/service/monitors/&lt;id&gt;/reset-failures</code>, MCP <code>service_monitor_reset_failures</code>, or <code>resetServiceMonitorFailures()</code> to clear only reviewed failures while keeping the last failure evidence. The runner updates <code>lastCheckedAt</code>, <code>lastSucceededAt</code>, <code>lastFailedAt</code>, <code>lastResult</code>, and <code>consecutiveFailures</code>; failed probes set the monitor <code>state</code> to <code>faulted</code> and append a service incident event.

Service control jobs do not time out at the worker boundary by default. Set <code>service.jobTimeoutMs</code>, pass <code>--service-job-timeout &lt;ms&gt;</code>, or set <code>AGENT_BROWSER_SERVICE_JOB_TIMEOUT_MS</code> to mark long-running dispatched jobs as <code>timed_out</code>. Use <code>0</code> to disable it. Jobs include advisory <code>namingWarnings</code> when a request is missing <code>serviceName</code>, <code>agentName</code>, or <code>taskName</code>; anonymous compatibility requests are still accepted.

Browser recovery defaults to 3 relaunch attempts, 1000 ms base backoff, and 30000 ms max backoff before marking a browser <code>faulted</code>. Set <code>service.recoveryRetryBudget</code>, <code>service.recoveryBaseBackoffMs</code>, and <code>service.recoveryMaxBackoffMs</code>, pass the matching service recovery CLI flags, or use the <code>AGENT_BROWSER_SERVICE_RECOVERY_*</code> environment variables to tune this for a service host. Recovery-started trace events include <code>details.policySource.retryBudget</code>, <code>details.policySource.baseBackoffMs</code>, and <code>details.policySource.maxBackoffMs</code> so clients can audit whether each active value came from defaults, config, environment, or CLI flags.

When runtime-profile config marks a service with <code>manualLoginPreferred</code>,
navigation to known login hosts for that service returns an advisory warning.
Use that warning as the signal to run <code>runtime login</code>, or
for ordinary sites, <code>runtime login --attachable</code> followed by
<code>runtime attach</code>. For Google, Gmail, and similar SSO flows, sign in
with detached <code>runtime login</code>, close Chrome, then relaunch the same
profile with <code>--attachable</code> for automation.
Use <code>--leave-open</code> when you want <code>close</code> to detach from a
managed runtime-profile browser instead of shutting it down.
Runtime profiles can declare <code>browserFamily</code>. agent-browser refuses
to launch a marked profile with a different resolved browser family unless the
operator sets <code>AGENT_BROWSER_ALLOW_PROFILE_BROWSER_MISMATCH=true</code>.

existing login state, inspect <code>service status</code> or
<code>runtime status</code>, then reuse the managed browser through
service/session control or attach to the intended runtime profile. Use
<code>--profile &lt;path&gt;</code> only for separate browser identities or
unauthenticated throwaway work.
Profile-lock errors include compact <code>diagnostic=&#123;...&#125;</code>
JSON with the lock PID, matching runtime profile, matching service
browser/session when known, and safe reuse, close, inspect, or separate-profile
remedies.
Normal launch commands automatically reuse a selected managed runtime browser
when agent-browser runtime state shows a live PID and reachable DevTools port.
<code>runtime status</code> reports <code>DevTools reachable</code> so stale
runtime-state ports are visible before an attach attempt.

## Batch execution

Execute multiple commands in a single invocation. Commands can be passed as quoted arguments or piped as JSON via stdin.

```bash
# Argument mode: each quoted argument is a full command
agent-browser batch "open https://example.com" "snapshot -i" "screenshot"

# With --bail to stop on first error
agent-browser batch --bail "open https://example.com" "click @e1" "screenshot"

# One daemon queue lease with target identity checks and step timings
agent-browser batch --dependent --bail "click @e1" "wait 1000" "screenshot"

# Stdin mode: pipe commands as JSON
echo '[
  ["open", "https://example.com"],
  ["snapshot", "-i"],
  ["click", "@e1"],
  ["screenshot", "result.png"]
]' | agent-browser batch --json
```

<table>
  <thead>
    <tr><th>Option</th><th>Description</th></tr>
  </thead>
  <tbody>
    <tr><td><code>--bail</code></td><td>Stop on first error (default: continue all commands)</td></tr>
    <tr><td><code>--dependent</code></td><td>Run parsed steps under one daemon queue lease with target identity checks and structured timings</td></tr>
    <tr><td><code>--json</code></td><td>Output results as a JSON array</td></tr>
  </tbody>
</table>

With <code>--dependent</code>, every step is parsed before execution. Stable
steps fail if active target identity changes, while navigation and tab commands
rebind the following step. Nested batches and daemon lifecycle actions are
rejected. The response includes queue wait, command preparation, action,
serialization, daemon total, and overall total timings. Existing batch behavior
is unchanged when the option is absent.

## Command chaining

Chain commands with `&&` in a single shell invocation. The browser persists via a background daemon, so chaining works naturally and is more efficient than separate calls:

```bash
agent-browser open example.com && agent-browser wait --load networkidle && agent-browser snapshot -i
agent-browser fill @e1 "user@example.com" && agent-browser fill @e2 "pass" && agent-browser click @e3
agent-browser open example.com && agent-browser wait --load networkidle && agent-browser screenshot page.png
```

Use `&&` when you don't need to read intermediate output. Run commands separately when you need to parse output first (e.g., snapshot to discover refs, then interact with those refs).
