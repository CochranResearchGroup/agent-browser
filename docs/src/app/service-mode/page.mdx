# Service Mode

Service mode turns agent-browser from a single CLI session helper into a persistent browser control plane. It keeps service-owned state for profiles, sessions, browsers, tabs, jobs, incidents, events, site policies, providers, and challenges so multiple agents and software clients can share browser automation without racing each other.

The model is intentionally broader than the CLI. Agents usually interact through MCP, software projects usually interact through the HTTP API, and operators can inspect or intervene through the CLI and dashboard.

The normal request model is identity-first. A caller asks for a tab or browser action plus `serviceName`, `agentName`, `taskName`, and the target site or login identity. agent-browser selects or reuses the managed profile, manages any profile lease, queues CDP work, and records the trace state. Direct profile selection is for known-login overrides and bring-your-own-profile workflows.

## What service mode provides

- A queued worker path for browser-control actions so concurrent clients are dispatched in order instead of competing directly for CDP.
- Persistent service records under the service state file, plus configured service records from `agent-browser.json` and `~/.agent-browser/config.json`.
- Trace context fields for `serviceName`, `agentName`, and `taskName`, so a service such as `JournalDownloader` can distinguish a task such as `probeACSwebsite`.
- Browser health detection for process exits, CDP disconnects, unreachable endpoints, target discovery failures, shutdown failures, and successful recovery.
- Operator remedies for job cancellation, incident acknowledgement or resolution, browser retry overrides, service reconciliation, polite browser close, and forced kill fallback.
- API and MCP parity for service records and typed browser-control actions.

## Service clients

Use the CLI when a human or shell script needs a direct operational view:

```bash
agent-browser service status
agent-browser service watch
agent-browser service trace --service-name JournalDownloader --task-name probeACSwebsite
agent-browser service incidents --limit 20
agent-browser service remedies
agent-browser service remedies apply --escalation monitor_attention --by operator --note reviewed
agent-browser service remedies apply --escalation browser_degraded --by operator --note reviewed
agent-browser service remedies apply --escalation os_degraded_possible --by operator --note host-inspected
agent-browser service jobs --limit 20
agent-browser service reconcile
```

Use the HTTP API when another program owns the workflow lifecycle:

```bash
curl "http://127.0.0.1:<stream-port>/api/service/status"
curl "http://127.0.0.1:<stream-port>/api/service/trace?serviceName=JournalDownloader&taskName=probeACSwebsite"
curl -X POST "http://127.0.0.1:<stream-port>/api/browser/navigate" \
  -H "content-type: application/json" \
  -d '{"url":"https://example.com","serviceName":"JournalDownloader","agentName":"article-probe-agent","taskName":"probeACSwebsite"}'
```

For software clients, prefer one explicit service request object when the caller wants agent-browser to select the profile, serialize CDP access, and retain trace metadata:

```ts
import { createServiceRequest, postServiceRequest, requestServiceTab } from '@agent-browser/client/service-request';

const request = createServiceRequest({
  serviceName: 'JournalDownloader',
  agentName: 'article-probe-agent',
  taskName: 'probeACSwebsite',
  siteId: 'acs',
  action: 'navigate',
  params: {
    url: 'https://example.com',
    waitUntil: 'load',
  },
  jobTimeoutMs: 30000,
});

const result = await postServiceRequest({
  baseUrl: `http://127.0.0.1:${streamPort}`,
  request,
});

const tab = await requestServiceTab({
  baseUrl: `http://127.0.0.1:${streamPort}`,
  serviceName: 'JournalDownloader',
  agentName: 'article-probe-agent',
  taskName: 'probeACSwebsite',
  siteId: 'acs',
  loginId: 'acs',
  url: 'https://example.com',
  jobTimeoutMs: 30000,
});
```

Use MCP when an agent runtime wants resources and tools without scraping CLI text:

```bash
agent-browser mcp serve
agent-browser mcp resources
agent-browser mcp read agent-browser://browsers
agent-browser mcp read agent-browser://jobs
```

An MCP client sends the same intent shape through `tools/call`:

```json
{
  "name": "service_request",
  "arguments": {
    "serviceName": "JournalDownloader",
    "agentName": "article-probe-agent",
```

MCP tool callers should include `serviceName`, `agentName`, and `taskName` whenever those values are known. Anonymous callers remain compatible, but service jobs record `namingWarnings` so operators can find unlabelled automation.

Retained service jobs also preserve target identity hints. Singular <code>targetServiceId</code>, <code>siteId</code>, and <code>loginId</code> fields keep the exact singular request values when present, and <code>targetServiceIds</code> stores the normalized target-service, site, and login identity set used by profile selection. Trace and job clients can show this profile-selection context without replaying the original command payload.

Trace summary context rows expose that same normalized identity as <code>targetServiceIds</code> with <code>targetIdentityCount</code>, so dashboards can display the site or login lane for each service, agent, task, browser, profile, or session context without scanning raw job arrays.

The HTTP service request object follows <code>docs/dev/contracts/service-request.v1.schema.json</code>. The MCP <code>tools/call</code> wrapper follows <code>docs/dev/contracts/service-request-mcp-tool-call.v1.schema.json</code>. HTTP <code>GET /api/service/contracts</code> and MCP <code>agent-browser://contracts</code> expose the same compatibility metadata for schema IDs, contract versions, routes, MCP tools, and supported service-request actions. The HTTP contracts metadata also names the <code>@agent-browser/client/service-observability</code> helpers for profile readiness, profile lookup, and access-plan responses, including the profile lookup selection order. The private workspace package <code>@agent-browser/client</code> generates its service request helpers and types from those schemas, including <code>requestServiceTab</code> for the common software-client path where a service asks for a queued tab by site or login identity. <code>postServiceRequest</code> and <code>requestServiceTab</code> return <code>ServiceRequestResponse</code>, the standard service command envelope with <code>success</code>, optional <code>data</code>, optional <code>error</code>, optional <code>warning</code>, and action-specific fields. Every action currently listed in <code>docs/dev/contracts/service-request.v1.schema.json</code> gets a typed <code>data</code> shape through <code>ServiceRequestDataForAction</code>; <code>requestServiceTab</code> returns typed <code>tab_new</code> data. Service request action changes must keep Rust <code>SERVICE_REQUEST_ACTIONS</code>, the JSON schema enum, MCP <code>service_request</code>, HTTP <code>/api/service/request</code>, and generated client files aligned. Service request helpers preserve <code>loginId</code> and <code>targetServiceId</code> in tab requests before the request is sent, and accept <code>profileLeasePolicy: "reject"</code> or <code>"wait"</code> plus <code>profileLeaseWaitTimeoutMs</code> for callers that want agent-browser to wait for a profile lease rather than fail immediately. Run <code>pnpm test:service-client</code> after client schema or helper changes to validate generated files, types, package export resolution, typed service command responses, service request helpers, and observability helpers.

With <code>profileLeasePolicy: "wait"</code>, the control-plane scheduler keeps the blocked request queued while it polls for profile release, so the worker can continue dispatching unrelated service requests.

Due active service monitors are also scheduled through the service worker. The daemon checks for due monitors every 60000 ms by default; set <code>service.monitorIntervalMs</code>, pass <code>--service-monitor-interval &lt;ms&gt;</code>, or set <code>AGENT_BROWSER_SERVICE_MONITOR_INTERVAL_MS</code> to change that interval. Use <code>0</code> to disable monitor scheduling. Use CLI <code>service monitors run-due</code>, HTTP <code>POST /api/service/monitors/run-due</code>, MCP <code>service_monitors_run_due</code>, or <code>runDueServiceMonitors()</code> to check due active monitors immediately. Use pause/resume commands, HTTP <code>POST /api/service/monitors/&lt;id&gt;/pause</code> and <code>POST /api/service/monitors/&lt;id&gt;/resume</code> routes, MCP tools, or client helpers to quiet and restore noisy monitors without clearing retained health history. Use triage commands, HTTP <code>POST /api/service/monitors/&lt;id&gt;/triage</code>, MCP <code>service_monitor_triage</code>, or <code>triageServiceMonitor()</code> to acknowledge the related monitor incident and clear reviewed failures in one queued operation. Use reset commands, HTTP <code>POST /api/service/monitors/&lt;id&gt;/reset-failures</code>, MCP <code>service_monitor_reset_failures</code>, or <code>resetServiceMonitorFailures()</code> after triage to clear only the reviewed failure counter while keeping the last failure evidence. Use CLI <code>service remedies apply --escalation monitor_attention</code>, HTTP <code>POST /api/service/remedies/apply?escalation=monitor_attention</code>, MCP <code>service_remedies_apply</code>, or <code>applyServiceRemedies()</code> to apply all active monitor remedies in one serialized operation. Use <code>service remedies apply --escalation browser_degraded</code> after operator review to batch retry enablement for active degraded-browser incidents. Use <code>service remedies apply --escalation os_degraded_possible</code> only after host inspection to batch the existing faulted-browser retry remedy for active OS-degraded-possible incidents. Monitor checks update <code>lastCheckedAt</code>, <code>lastSucceededAt</code>, <code>lastFailedAt</code>, <code>lastResult</code>, and <code>consecutiveFailures</code>; failed probes set the monitor <code>state</code> to <code>faulted</code> and append a service incident event. CLI <code>service monitors</code> and HTTP <code>GET /api/service/monitors</code> support state, failed-only, and summary filters so operators and software clients can find repeated failures without scanning raw events. MCP <code>agent-browser://monitors</code> remains the full unfiltered retained resource. Monitor incidents use <code>monitor_attention</code>, expose <code>monitorId</code>, <code>monitorTarget</code>, and <code>monitorResult</code>. Incident summary groups include <code>browserIds</code>, <code>monitorIds</code>, <code>monitorResetCommands</code>, and <code>remedyApplyCommand</code> so operators and agents can see the exact affected browsers and the batch apply command.

Read-side and service-configuration software clients can import <code>@agent-browser/client/service-observability</code> for generated helpers around service status, service contract metadata, service collections, config mutations, operator remedies, reconcile, jobs, events, incidents, incident activity, and trace responses. Collection helpers cover profiles, browsers, sessions, tabs, monitors, site policies, providers, and challenges. <code>getServiceMonitors</code> reads retained monitor records for heartbeat, tab, site-policy, and login freshness probes, and accepts <code>state</code>, <code>failedOnly</code>, and <code>summary</code> options for software clients that need compact monitor triage. Active monitors are checked by the daemon scheduler when due. <code>getServiceProfileAllocation</code> reads one profile's current allocation row, <code>getServiceProfileReadiness</code> reads one profile's no-launch target readiness, <code>summarizeServiceProfileReadiness</code> turns readiness rows into client-facing manual-seeding flags and recommended actions, <code>findServiceProfileForIdentity</code> applies the same authenticated-target, target-match, then service-allow-list profile preference order to a local profile array, and <code>lookupServiceProfile</code> calls HTTP <code>GET /api/service/profiles/lookup</code> so agent-browser applies the authoritative service selector and returns the selected profile, selector reason, matched profile field, matched identity, readiness, and readiness summary without returning the full profile collection. <code>getServiceAccessPlan</code> calls HTTP <code>GET /api/service/access-plan</code> and returns the no-launch service-owned recommendation that combines selected profile, readiness, site policy, enabled providers, retained challenges, caller label warnings, and decision fields before a caller asks for browser control. <code>getServiceProfileForIdentity</code> remains as the older descriptive alias for the same route. Config mutation helpers upsert and delete profiles, sessions, site policies, monitors, and providers. <code>upsertServiceMonitor</code> and <code>deleteServiceMonitor</code> call HTTP <code>POST /api/service/monitors/&lt;id&gt;</code> and <code>DELETE /api/service/monitors/&lt;id&gt;</code> to persist monitor definitions for the scheduler. <code>getServiceContracts</code> reads the compatibility metadata endpoint. <code>registerServiceLoginProfile</code> wraps the common software-client recipe for registering a profile by login identity and can record bounded-probe freshness with <code>readinessState</code>, <code>readinessEvidence</code>, <code>lastVerifiedAt</code>, and <code>freshnessExpiresAt</code>. <code>updateServiceProfileFreshness</code> posts to the service-side freshness mutation endpoint, which merges the new readiness row under the serialized service-state mutator, preserves unrelated fields, and updates <code>authenticatedServiceIds</code> for fresh, stale, or blocked target states. Operator remedy helpers cancel jobs, retry browsers, apply service remedies with <code>applyServiceRemedies()</code>, and acknowledge or resolve incidents.

Software clients should ask agent-browser for an access plan before creating new runtime state:

```ts
import { requestServiceTab } from '@agent-browser/client/service-request';
import {
  getServiceAccessPlan,
  registerServiceLoginProfile,
} from '@agent-browser/client/service-observability';

const baseUrl = `http://127.0.0.1:${streamPort}`;
const serviceName = 'CanvaCLI';
const loginId = 'canva';

const accessPlan = await getServiceAccessPlan({
  baseUrl,
  serviceName,
  agentName: 'canva-cli-agent',
  taskName: 'openCanvaWorkspace',
  loginId,
  targetServiceId: loginId,
});

if (!accessPlan.selectedProfile) {
  await registerServiceLoginProfile({
    baseUrl,
    id: 'canva-default',
    serviceName,
    loginId,
    authenticated: false,
  });
}

await requestServiceTab({
  baseUrl,
  serviceName,
  agentName: 'canva-cli-agent',
  taskName: 'openCanvaWorkspace',
  loginId,
  targetServiceId: loginId,
  url: 'https://www.canva.com/',
  jobTimeoutMs: 30000,
});
```

If <code>accessPlan.readinessSummary.needsManualSeeding</code> is true, show the returned recommended actions to the operator and seed the managed profile before expecting authenticated automation to succeed for the requested identity. Access-plan readiness decisions are scoped to the requested target identities, so an unrelated stale or unseeded login on the same profile does not block the requested site. Access-plan responses echo <code>agentName</code> and <code>taskName</code> in <code>query</code>, and both <code>query</code> and <code>decision</code> include <code>namingWarnings</code> plus <code>hasNamingWarning</code> when <code>serviceName</code>, <code>agentName</code>, or <code>taskName</code> is missing. The decision also separates auth providers from challenge-capable providers, reports the challenge strategy, lists missing provider capabilities such as <code>captcha_solve</code>, <code>sms_code</code>, or <code>human_approval</code>, includes <code>interactionRisk</code> plus a <code>pacing</code> block derived from site-policy rate limits, exposes <code>launchPosture</code> so clients can see whether the resolved browser host is headed, remote-view capable, or requires detached first-login seeding, and includes <code>freshnessUpdate</code> with the selected profile, target identities, HTTP route, MCP tool, and <code>updateServiceProfileFreshness</code> helper to use after a bounded auth probe. When no local site policy exists, agent-browser applies shipped defaults for Google, Gmail, and Microsoft login identities. Local persisted or configured policies with the same IDs override those defaults, and <code>sitePolicySource</code> reports the selected policy source, match path, precedence, and whether the policy can be overridden.

When a software client has just completed a bounded auth probe, pass <code>readinessState: "fresh"</code>, <code>readinessEvidence</code>, <code>lastVerifiedAt</code>, and <code>freshnessExpiresAt</code> to <code>registerServiceLoginProfile()</code>. For an already registered profile, call <code>updateServiceProfileFreshness()</code> so the service merges the new row under the serialized service-state mutator, preserves unrelated fields, and removes stale or blocked targets from <code>authenticatedServiceIds</code>. The service preserves explicit <code>fresh</code>, <code>stale</code>, and <code>blocked_by_attached_devtools</code> evidence through derived readiness refreshes.

The repo includes <code>examples/service-client/</code> as a copyable software-client workflow. It requests a service tab with <code>requestServiceTab</code>, reads <code>getServiceTrace</code>, can cancel a known queued job with <code>cancelServiceJob</code>, has a dry-run check through <code>pnpm test:service-client-example</code>, and has a live isolated daemon check through <code>pnpm test:service-client-example-live</code>. The same example package includes <code>managed-profile-flow.mjs</code>, a CanvaCLI-style profile-broker recipe that uses <code>getServiceAccessPlan</code> to inspect the selected profile, readiness, site policy, providers, retained challenges, and service-owned decision before requesting a tab by login identity or registering a fallback managed profile. It can also post bounded auth-probe evidence through <code>updateServiceProfileFreshness()</code> for an existing managed profile. Its output includes <code>readinessSummary.needsManualSeeding</code> plus target service IDs and recommended actions when readiness says an operator must seed the profile. Run <code>pnpm test:service-client-managed-profile-flow</code> for the no-launch mock smoke that proves the recipe selects an existing managed profile without registering a new one.

## Operator request model

agent-browser owns browsers, profiles, tabs, queues, and job dispatch by default. Operators, agents, and software clients should usually request intent instead of taking direct responsibility for browser coordination. A normal request says which site or login identity is needed, what tab or browser action should happen, and which service, agent, and task owns the work. agent-browser then selects or reuses the right managed profile, attaches to an existing managed browser when possible, queues CDP control deterministically, and records the state needed for debugging. Profile records and profile allocation rows include `targetReadiness`, a no-launch readiness view for target services. Profile collections include `profileSources`, and profile lookup plus access-plan responses include `selectedProfileSource`, so operators can see whether the selected effective profile came from config, runtime observation, or persisted state. Google targets without authenticated evidence report `needs_manual_seeding` and recommend detached `runtime login` before attachable automation. Once a managed profile lists the target in `authenticatedServiceIds`, readiness changes to `seeded_unknown_freshness` and access-plan no longer treats first-login seeding as a required manual action. HTTP clients can read one profile's readiness rows from `GET /api/service/profiles/<id>/readiness` or ask agent-browser to select by service and identity through `GET /api/service/profiles/lookup`.

Use `serviceName`, `agentName`, `taskName`, and target-service hints such as `targetServiceId`, `targetService`, `targetServiceIds`, or `targetServices` to make this request explicit. Callers that think in site or login vocabulary can use `siteId`, `siteIds`, `loginId`, or `loginIds`; agent-browser normalizes those aliases into the same profile-selection request. A service named `JournalDownloader` can request an ACS-authenticated tab for `probeACSwebsite`; agent-browser should decide whether the current ACS-authenticated profile and browser can be shared, whether a service-specific profile is required, or whether a new browser process is justified by isolation policy.

Operators can still request a particular managed runtime profile when they know the desired login state lives there. They can also bring their own profile with `--profile <path>` or equivalent API fields when they want more direct control over a browser identity. That is an override mode, not the default service model. The normal path is to let agent-browser perform deterministic profile selection, queue sharing, browser reuse, and collision avoidance so callers can spend their limited reasoning budget on the target workflow.

Software clients should not create a runtime profile just because another task
is active or because a site has a login. First call
<code>getServiceAccessPlan()</code> for the target identity, then request the
tab with <code>requestServiceTab</code> or <code>POST /api/service/request</code>.
Use <code>getServiceProfiles()</code> or
<code>getServiceProfileReadiness(&#123; id &#125;)</code> for narrower inspection
when a caller already knows which profile it is evaluating. Register a new
managed login profile only when agent-browser has no suitable profile, readiness
reports <code>needs_manual_seeding</code>, the operator wants a separate account
lane, or the client is explicitly bringing its own profile.

## Login Identity API Recipe

Software clients should request the identity they need, not the browser process they hope is free. Register the profile once with the target identities it can serve:

```ts
await registerServiceLoginProfile({
  baseUrl: `http://127.0.0.1:${streamPort}`,
  id: 'journal-acs',
  serviceName: 'JournalDownloader',
  loginId: 'acs',
});
```

Then request a tab by login identity through the service client:

```ts
await requestServiceTab({
  baseUrl: `http://127.0.0.1:${streamPort}`,
  serviceName: 'JournalDownloader',
  agentName: 'article-probe-agent',
  taskName: 'probeACSwebsite',
  loginId: 'acs',
  url: 'https://example.com',
});
```

agent-browser first prefers profiles whose <code>authenticatedServiceIds</code> match the requested identity, then profiles whose <code>targetServiceIds</code> match, then profiles shared with the calling service through <code>sharedServiceIds</code>. Session records expose <code>profileSelectionReason</code> as <code>authenticated_target</code>, <code>target_match</code>, <code>service_allow_list</code>, or <code>explicit_profile</code>, and launch events mirror that value in <code>details.profileSelectionReason</code>. They also expose <code>profileLeaseDisposition</code> as <code>new_browser</code>, <code>reused_browser</code>, or <code>active_lease_conflict</code> plus <code>profileLeaseConflictSessionIds</code> when another exclusive session already references the same profile. Service-scoped launches reject active exclusive profile conflicts by default before browser start; set <code>profileLeasePolicy: "wait"</code> and <code>profileLeaseWaitTimeoutMs</code> to wait for release instead. Same-session retained browser reuse remains allowed. Use explicit <code>profile</code> or <code>runtimeProfile</code> request fields only for override workflows where the caller intentionally takes responsibility for the browser identity.

For example, CanvaCLI should request <code>serviceName: "CanvaCLI"</code> plus
<code>loginId: "canva"</code> or <code>targetServiceId: "canva"</code>. If no
suitable managed Canva profile exists, register one with
<code>registerServiceLoginProfile(&#123; id: "canva-default", serviceName: "CanvaCLI", loginId: "canva", authenticated: false &#125;)</code>,
seed it when readiness says manual seeding is needed, and keep future work
identity-based instead of profile-name-based.

HTTP <code>GET /api/service/profiles/&lt;id&gt;/allocation</code> returns one derived profile allocation row when a software client wants current holder, waiting job, conflict, lease state, and recommended-action data without fetching the full profile collection.

Waiting lease requests stay queued until the selected profile becomes available or <code>profileLeaseWaitTimeoutMs</code> expires.

## Profile and session planning

Runtime profiles are the safest default for recurring automation because they keep Chrome user data, cookies, site storage, and login state in a managed path. A single profile can represent one user identity per site, so profile allocation is a service design decision:

- Use one shared profile when several tasks intentionally operate as the same browser identity.
- Use a service-specific profile when a service needs durable login state but should not share cookies or extensions with unrelated services.
- Prefer the profile that has credentials and usable auth state for the target site or identity provider.
- Use a separate browser process when profile isolation, keyring behavior, extension state, or resource ownership matters more than process count.
- Use a custom `--profile <path>` only when an external profile directory is part of the contract.

Configured service profiles and sessions can describe ownership, lease behavior, cleanup policy, keyring posture, browser linkage, caller labels, intended target services, currently believed authenticated target services, the reason a profile was selected, and whether the selected profile opened a new browser, reused a retained browser, or overlapped another exclusive session. `targetServiceIds` records target sites or identity providers such as `google`, `microsoft`, or `acs`; `authenticatedServiceIds` records targets currently believed to have usable login state. Profile writes reject `caller_supplied` records without `userDataDir` and `per_service` records with multiple `sharedServiceIds` values. Session writes infer `owner` from `agentName`, then `serviceName`, when omitted; `profileId` must reference a persisted profile, and profile `sharedServiceIds` allow-lists are enforced. Explicit `--runtime-profile` and `--profile` values still win. When a launch command omits both, `serviceName` plus `targetServiceId`, `targetService`, `targetServiceIds`, `targetServices`, `siteId`, `siteIds`, `loginId`, or `loginIds` lets agent-browser choose a persisted service profile. The selector first prefers `authenticatedServiceIds` matches, then `targetServiceIds` matches, then the caller `sharedServiceIds` match. MCP typed browser tools and `service_request` accept the same target profile hints, so clients can stay on typed schemas or send one explicit service request object for profile-aware first commands. Run `pnpm test:service-profile-target-mcp-live` to validate this live MCP path.

## Browser health and recovery

Service reconciliation probes persisted browser records and updates health before clients make decisions from stale state. Local process exits are marked `process_exited`. CDP endpoints with a live process but failed CDP connection are marked `cdp_disconnected`. Endpoints without a live process are marked `unreachable`. Endpoints that answer health probes but fail target discovery are marked `degraded`.

When a queued browser command finds that the active browser has exited or CDP is disconnected, agent-browser records the failure before cleanup and relaunch. Recovery traces show the health transition, recovery start, retry budget, policy source, next retry delay, and the final ready transition when relaunch succeeds.

Close handling uses two remedies. First, agent-browser attempts a polite browser shutdown. If polite shutdown fails, the browser record is degraded. Then it force kills owned browser processes when needed. If force kill fails, the browser record is faulted and the incident can escalate as possible OS degradation.

Run `pnpm test:service-shutdown-faulted-live` to validate the faulted shutdown path.

## Service records

The CLI, HTTP API, MCP resources, and dashboard share the same service concepts:

- `profiles` describe durable browser identity and launch policy.
- `sessions` describe service or agent ownership of browser work.
- `browsers` describe process, CDP, health, and recovery state.
- `tabs` describe live page and webview targets discovered from CDP.
- `jobs` describe queued, waiting-on-profile-lease, running, succeeded, failed, cancelled, and timed-out service work.
- `controlPlane.waitingProfileLeaseJobCount` summarizes current profile lease wait pressure without requiring a separate jobs query.
- `incidents` group operational issues with severity, escalation, handling state, and recommended action.
- `events` record launch, reconciliation, health, recovery, profile lease wait, tab lifecycle, and incident-handling changes.
- `sitePolicies`, `providers`, and `challenges` describe site-specific automation posture, browser providers, and retained challenge state.

The repo keeps JSON schemas for these contracts under `docs/dev/contracts/`. HTTP and MCP collection envelopes are expected to remain aligned with those schemas.

```text
DELETE /api/service/site-policies/<id>
POST   /api/service/providers/<id>
DELETE /api/service/providers/<id>
```

Named browser-control routes accept the same action fields as the underlying command, plus trace fields such as `serviceName`, `agentName`, `taskName`, and `jobTimeoutMs`. `POST /api/service/request` accepts one intent object with `serviceName`, `agentName`, `taskName`, `siteId`, `loginId`, target-service hints, explicit `profile` or `runtimeProfile` hints, `action`, `params`, and `jobTimeoutMs`, then queues the requested browser action through the same service-owned control path. Its request object follows <code>docs/dev/contracts/service-request.v1.schema.json</code>.

`agent-browser mcp serve` exposes read-only service resources and service/browser tools over stdio. Important resources include:

```text

agent-browser://incidents/{incident_id}/activity
```

Service tools include `service_request`, `service_job_cancel`, `service_browser_retry`, `service_incidents`, `service_remedies_apply`, `service_trace`, `service_profile_upsert`, `service_profile_freshness_update`, `service_profile_delete`, `service_session_upsert`, `service_session_delete`, `service_site_policy_upsert`, `service_site_policy_delete`, `service_monitor_upsert`, `service_monitor_delete`, `service_monitors_run_due`, `service_monitor_pause`, `service_monitor_resume`, `service_monitor_reset_failures`, `service_monitor_triage`, `service_provider_upsert`, and `service_provider_delete`. Use `service_request` to queue one intent-based browser action with caller context and site/login hints. Pass `summary: true` to `service_incidents` for the same escalation, severity, and state remedy groups as HTTP `summary=true`; add `remediesOnly: true` for the compact active `browser_degraded`, `monitor_attention`, and `os_degraded_possible` operator ladder. Use `service_remedies_apply` with `escalation: "monitor_attention"` to apply active monitor remedies, `escalation: "browser_degraded"` after operator review to batch degraded-browser retry enablement, or `escalation: "os_degraded_possible"` after host inspection to batch faulted-browser retry remedies through the service worker.

Typed browser tools cover navigation, tabs, snapshots, screenshots, clicks, fills, waits, keyboard actions, element reads, cookies, storage, headers, offline mode, viewport, geolocation, permissions, timezone, locale, media, dialogs, uploads, downloads, HAR capture, request routing, console, errors, PDFs, response bodies, and clipboard operations. Use `browser_command` only for browser actions that do not yet have a typed tool.

## Dashboard operations

The dashboard service view is the operator surface for long-lived automation. It should organize multiple services and agents at the same time, show named tasks, expose service traces and incidents, and make browser state inspectable without requiring every operator to attach directly to Chrome.

When live browser viewing is enabled, use it as an inspection and intervention surface, not as the source of truth. Service records, trace entries, and job or incident state remain the durable control-plane state.

## Pre-release validation

Run the release-facing documentation and service contract checks before cutting a release that includes service mode:

```bash
pnpm test:service-api-mcp-parity
pnpm test:service-contracts-no-launch
pnpm test:service-profile-lookup-no-launch
pnpm test:service-profile-sources-no-launch
pnpm test:service-incident-summary-http
pnpm test:service-incident-summary-mcp
pnpm test:service-request-live
pnpm test:service-profile-lease-wait-live
cd docs && pnpm build
git diff --check
```

Ordinary fast CI runs the no-launch service contract metadata smoke, the no-launch profile-source smoke, the no-launch site-policy source smoke, and the HTTP and MCP incident-summary smokes after the Rust suite, so service compatibility metadata, effective profile and site-policy provenance, and grouped service incident remedies stay covered without starting Chrome. Service request action changes are guarded by the static API/MCP parity check, generated client contract check, and targeted Rust MCP and HTTP action-parity tests. Run both incident-summary smokes before changing incident summary grouping or filters; together they guard HTTP `summary=true` and MCP `service_incidents` with `summary: true` across state, severity, escalation, handling-state, browser, profile, session, service, agent, task, and since filters.

Run the broader service health smoke when Chrome-backed live validation is required:

```bash
pnpm test:service-health-live
```

Run the profile lease wait smoke when service request, profile selection, lease contention, trace summary, dashboard trace, or service observability client behavior changes:

```bash
pnpm test:service-profile-lease-wait-live
```

Run the Rust service tests when the release includes service implementation changes:

