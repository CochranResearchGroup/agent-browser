# Configuration

Create an `agent-browser.json` file to set persistent defaults instead of repeating flags on every command.

## Config File Locations

agent-browser checks two locations, merged in priority order:

<table>
  <thead>
    <tr><th>Priority</th><th>Location</th><th>Scope</th></tr>
  </thead>
  <tbody>
    <tr><td>1 (lowest)</td><td><code>~/.agent-browser/config.json</code></td><td>User-level defaults</td></tr>
    <tr><td>2</td><td><code>./agent-browser.json</code></td><td>Project-level overrides</td></tr>
    <tr><td>3</td><td><code>AGENT_BROWSER_*</code> env vars</td><td>Override config values</td></tr>
    <tr><td>4 (highest)</td><td>CLI flags</td><td>Override everything</td></tr>
  </tbody>
</table>

Project-level values override user-level values. Environment variables override both. CLI flags always win.

Use `--config <path>` or the `AGENT_BROWSER_CONFIG` environment variable to load a specific config file instead of the default locations:

```bash
agent-browser --config ./ci-config.json open example.com
AGENT_BROWSER_CONFIG=./ci-config.json agent-browser open example.com
```

If you do not set <code>profile</code> or <code>runtimeProfile</code>, agent-browser uses the managed default runtime profile at <code>~/.agent-browser/runtime-profiles/default/user-data</code>. Set <code>runtimeProfile</code> for a named managed profile, or <code>profile</code> for a custom persistent user-data-dir path.

```json
  "runtimeProfiles": {

      "launch": {
        "headed": true,
        "leaveOpen": true,
        "proxy": "http://localhost:8080"
      },
      "auth": {
        "sessionName": "work-session",
        "manualLoginPreferred": true
      },
      "services": {
        "google": {
          "manualLoginPreferred": true
        }
      }
    }
  },
  "service": {
    "defaultBrowserBuild": "stealthcdp_chromium",
    "browserBuildManifests": {
      "stealthcdp_chromium": {
        "manifestPath": "/opt/agent-browser/chromium-stealthcdp/current/manifest.json"
      }
    },
    "browserCapabilityRegistry": {
      "browserHosts": [
        {
          "id": "local-linux",
          "name": "Local Linux desktop",
          "hostKind": "local",
          "operatingSystem": "linux",

          "tags": ["default"]
        }
      ],
      "browserExecutables": [],
      "browserCapabilities": [],
      "profileCompatibility": [],
      "browserPreferenceBindings": [],
      "validationEvidence": []
    },
    "reconcileIntervalMs": 60000,
    "jobTimeoutMs": 120000,
    "profiles": {
      "work": {
        "name": "Work",
        "allocation": "per_service",
        "keyring": "basic_password_store",
        "browserBuild": "stealthcdp_chromium",
        "sharedServiceIds": ["JournalDownloader"],
        "manualLoginPreferred": true
      }
    },

        "cleanup": "close_tabs"
      }
    }
  },
  "userAgent": "my-agent/1.0",
  "ignoreHttpsErrors": true
}
```

Runtime profiles are the preferred way to define durable managed browser lanes.
Today agent-browser applies the selected runtime profile's `userDataDir`,
browser family, launch defaults, auth session name, and service login hints. A service with
`manualLoginPreferred` emits an advisory warning when navigation targets known
login hosts for that service. Use this to tell agents to prefer
detached <code>runtime login</code> for services such as Google. Use the
attachable manual-login flow only for sites where DevTools during login is
accepted. Set <code>launch.leaveOpen</code> or pass
<code>--leave-open</code> when you want <code>close</code> to detach from a
managed runtime-profile browser instead of shutting it down. Set
<code>preferences.defaultViewport</code> to a <code>WIDTHxHEIGHT</code> value,
such as <code>960x640</code>, when a runtime profile should resize the browser
content area after launch and before the requested command runs.

Set <code>runtimeProfiles.&lt;name&gt;.browserFamily</code> to <code>chrome</code>,
<code>chromium</code>, <code>brave</code>, <code>edge</code>, or
<code>unknown</code> to bind a managed profile to the browser family that
created it. agent-browser refuses a different resolved browser family by
default because Chrome and Chromium profile directories are not safe to
mix-and-match. Start with a blank profile for a new browser family. Operators
can force an intentional mismatch with
<code>AGENT_BROWSER_ALLOW_PROFILE_BROWSER_MISMATCH=true</code>.

The <code>service.profiles</code> and <code>service.sessions</code> maps define
service control-plane metadata for profile allocation, keyring posture, caller
ownership, profile binding, lease state, and cleanup policy. These records are
exposed through service status, MCP resources, and the HTTP service APIs.
Set <code>service.defaultBrowserBuild</code> to
<code>stealthcdp_chromium</code> only when service-owned launches should prefer
the patched Chromium posture. If no explicit default is configured and a ready
<code>stealthcdp_chromium</code> manifest is available, fresh installs prefer
that build automatically. <code>agent-browser install stealthcdp-chromium</code>
installs the public <code>chromium-stealthcdp</code> Windows asset under
<code>%LOCALAPPDATA%\chromium-stealthcdp</code> on Windows or the matching
WSL-mounted <code>AppData/Local</code> directory when available, then exposes
<code>current/manifest.json</code> and <code>current/chrome.exe</code>.
When WSL launches that Windows <code>chrome.exe</code>, agent-browser translates
mounted Windows paths such as <code>/mnt/c/...</code> to <code>C:\...</code>
for Chrome arguments and adds <code>--no-sandbox</code>, which this host mode
requires for the patched Windows build to expose DevTools reliably.
Provide the current binary location with <code>executablePath</code>,
<code>AGENT_BROWSER_EXECUTABLE_PATH</code>,
<code>AGENT_BROWSER_STEALTHCDP_CHROMIUM_MANIFEST_PATH</code>,
<code>AGENT_BROWSER_STEALTHCDP_CHROMIUM_INSTALL_ROOT</code>, or
<code>service.browserBuildManifests.stealthcdp_chromium.manifestPath</code>.
A manifest path points at a promoted artifact <code>manifest.json</code>, and
agent-browser resolves the executable relative to that manifest before
launching.
<code>agent-browser service status</code> and
<code>GET /api/service/status</code> include a no-launch
<code>launchConfig</code> diagnostic with the resolved executable path,
executable source, artifact metadata, file existence, and warnings when the
selected custom build is not ready.
The patched build is useful because it keeps CDP-backed service control while
reducing the obvious automation signal exposed by ordinary DevTools-attached
Chromium. It is not a captcha bypass, but it is a better default for
bot-sensitive sites that still tolerate CDP-backed control.
Profiles can set <code>browserBuild</code> to <code>stealthcdp_chromium</code>,
<code>stock_chrome</code>, or <code>cdp_free_headed</code>. Exact authenticated
target, account, and target-site matches still win first; browser build then
breaks ties and can select a generic default profile for new identities. Use a
<code>stock_chrome</code> profile with target and account IDs for sites that
only work correctly in Google Chrome.
<code>service.browserCapabilityRegistry</code> carries draft browser host,
executable, capability, profile compatibility, preference binding, and
validation evidence records into
<code>service_state.browserCapabilityRegistry</code> for
<code>agent-browser service status --json</code> and
<code>GET /api/service/status</code>. The registry is advisory in this release
and does not affect profile selection or browser routing yet.
Use <code>agent-browser install doctor</code> after upgrading the user-scoped
command or changing browser build manifests. It does not launch Chrome. The
doctor checks the command on <code>PATH</code>, the running executable, the pnpm
global package binary when pnpm is available, the current workspace binary when
run from a checkout, and this same <code>launchConfig</code> readiness view.
Current browser launch behavior still comes from <code>runtimeProfiles</code>,
<code>--runtime-profile</code>, <code>--profile</code>, and existing launch
flags. Launches that select a runtime profile or custom profile path bind the
active browser record to a service profile. When commands include
<code>serviceName</code>, <code>agentName</code>, or <code>taskName</code>, the
active session record also captures that caller context for traceability.

Use <code>agent-browser runtime list</code> to inspect the merged runtime-profile
view from config plus on-disk managed profiles. If a profile sets
<code>userDataDir</code>, <code>runtime status</code> reports that configured path
even before Chrome has written a runtime state file.

Use <code>agent-browser runtime create [name] [--set-default]</code> when you want
to register a managed profile in <code>~/.agent-browser/config.json</code>
without editing JSON by hand.

## Overriding Boolean Options

Boolean flags accept an optional `true`/`false` value to override config settings:

```bash
agent-browser --headed false open example.com
```

A bare flag is equivalent to passing `true`:

```bash
agent-browser --headed open example.com       # same as --headed true
agent-browser --headed true open example.com  # explicit
```

This applies to all boolean flags: `--headed`, `--debug`, `--json`, `--ignore-https-errors`, `--allow-file-access`, `--auto-connect`, `--content-boundaries`, `--confirm-interactive`.

## Extensions Merging

Extensions from user-level and project-level configs are **concatenated**, not replaced. For example, if `~/.agent-browser/config.json` specifies `["/ext1"]` and `./agent-browser.json` specifies `["/ext2"]`, the result is `["/ext1", "/ext2"]`.

The `AGENT_BROWSER_EXTENSIONS` environment variable and CLI `--extension` flags follow the standard priority rules (env replaces config, CLI appends).

For local secret bootstrapping, agent-browser can also load `AGENT_BROWSER_ENV_FILE`, or `~/.agent-browser/.env` when that file exists. Environment variables from the real shell still win over the dotenv file.

## Error Handling

- **Auto-discovered config files** (`~/.agent-browser/config.json`, `./agent-browser.json`) that are missing are silently ignored.
- **`--config <path>`** with a missing or malformed file exits with an error.
- **Malformed JSON** in auto-discovered files prints a warning to stderr and continues without that file.
- **Unknown keys** are silently ignored for forward compatibility.

> **Tip:** If your project-level `agent-browser.json` contains environment-specific values (paths, proxies), consider adding it to `.gitignore`.
